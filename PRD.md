# AI 笔记管理应用 PRD

## 1. 产品概述

本产品是一款面向个人知识管理的桌面 AI 笔记应用。应用以本地 Markdown 文件为核心数据源，用户可以打开一个本地文件夹作为笔记库，直接浏览、编辑、搜索 Markdown 笔记，并使用 AI 对当前笔记或笔记库内容进行总结、改写、建议、问答和结构化整理。

产品的核心原则是：文件属于用户、AI 行为可审计、写入动作可撤回。第一版优先做一个稳定、可信、轻量的本地 Markdown + AI 工作台，而不是一开始做复杂的同步、插件系统或完整 Agent。

## 2. 目标用户

### 主要用户

- 使用 Markdown 记录学习、工作、项目、读书笔记的个人用户。
- 想要用 AI 总结、整理、改写、提取行动项，但不希望笔记被锁进私有数据库的用户。
- 需要桌面端稳定体验，并希望文件可被 Obsidian、Typora、VS Code 等工具继续打开的用户。

### 次要用户

- 研究人员、开发者、产品经理、学生、内容创作者。
- 希望未来接入本地模型或私有模型的隐私敏感用户。

## 3. 产品定位

产品定位为“带 AI 审阅和整理能力的本地 Markdown 笔记管理器”。

它不是：

- 不是云端 Notion 替代品。
- 不是一开始就做多人协作的知识库。
- 不是一开始就做完整自动化 Agent。
- 不是把 Markdown 内容全部迁移到数据库的封闭笔记系统。

它应该更接近 Obsidian/Typora 的本地文件体验，但 AI 写入更克制、更透明、更适合长期维护个人知识库。

## 4. 核心价值

1. 用户可以继续使用普通 Markdown 文件，避免数据锁定。
2. AI 可以理解当前笔记，并提供总结、改写、提取待办、标题建议、标签建议。
3. AI 对文件的修改必须经过用户确认，默认不直接覆盖笔记。
4. 应用能逐步扩展为全库问答、自动整理和本地模型能力。
5. 桌面端体验应快速、安静、稳定，适合长期打开使用。

## 5. 技术栈建议

### 推荐技术栈

- 桌面框架：Tauri
- 前端：React + TypeScript
- 后端：Rust
- 本地存储：SQLite
- 笔记格式：Markdown 文件
- AI 能力：第一版使用云端模型 API，后续可扩展本地模型

### 暂不建议加入 Python

第一版不建议引入 Python。Tauri + React + TypeScript + Rust 已足够实现桌面应用、文件读写、本地索引、AI API 调用和基础数据管理。

只有在后续明确需要以下能力时，再考虑通过 Tauri sidecar 引入 Python：

- 本地大模型推理编排。
- OCR、语音转写、复杂文档解析。
- 已有 Python AI pipeline 需要复用。
- 依赖 Python 生态的向量处理、机器学习或批处理任务。

提前加入 Python 会增加打包、跨平台发布、进程通信、错误排查和性能管理成本，不适合 MVP。

## 6. MVP 范围

### 必须实现

1. 打开本地文件夹作为笔记库。
2. 展示 Markdown 文件树。
3. 打开、编辑、保存 Markdown 文件。
4. 自动保存或明确保存状态提示。
5. 基础 Markdown 编辑体验。
6. 基础全文搜索。
7. AI 侧边栏，支持读取当前笔记内容。
8. AI 对当前笔记执行总结、提取待办、改写选中文本、建议标题、建议标签。
9. AI 写入前展示预览或 diff，用户确认后再应用。
10. 设置页，支持配置 API Key、模型和 AI 读取范围。
11. 本地 SQLite 保存索引、最近打开、摘要缓存、标签建议等元数据。

### 暂缓实现

- 云同步。
- 多人协作。
- 移动端。
- 插件市场。
- 完整知识图谱。
- 自动后台全库重写。
- 多 AI provider 复杂抽象。
- 本地大模型。
- 富文本块编辑器。

## 7. 核心用户流程

### 首次使用

1. 用户打开应用。
2. 应用提示选择一个本地文件夹作为笔记库。
3. 用户授权文件夹访问。
4. 应用扫描 `.md` 文件并建立基础索引。
5. 用户打开任意笔记开始编辑。

### AI 总结当前笔记

1. 用户打开一篇 Markdown 笔记。
2. 用户点击 AI 侧边栏中的“总结”。
3. AI 读取当前笔记内容。
4. AI 流式返回摘要。
5. 用户可以复制摘要，或选择插入到当前笔记。
6. 如果插入，应用先展示插入位置和预览内容，用户确认后写入。

### AI 改写选中文本

1. 用户在编辑器中选中一段文本。
2. 用户选择“改写”。
3. AI 根据选中文本和当前笔记上下文生成改写版本。
4. 应用展示原文和新文对比。
5. 用户确认后替换选中文本。

### AI 建议标签

1. 用户打开笔记。
2. 用户点击“建议标签”。
3. AI 根据笔记内容生成 3 到 8 个标签。
4. 用户选择要应用的标签。
5. 应用将标签写入 frontmatter 或应用元数据。

## 8. 功能需求

### 8.1 笔记库管理

- 支持选择本地文件夹作为 vault。
- 支持记住最近打开的 vault。
- 支持扫描子目录中的 Markdown 文件。
- 支持创建、重命名、删除 Markdown 文件。
- 支持创建文件夹。
- 删除文件必须二次确认，优先移动到系统废纸篓。

### 8.2 Markdown 编辑器

- 支持打开和编辑 `.md` 文件。
- 支持编辑状态提示：已保存、保存中、未保存、保存失败。
- 支持基础 Markdown 语法高亮。
- 支持预览或分屏预览，MVP 可只做编辑模式加简单预览。
- 支持选中文本后触发 AI 操作。

### 8.3 搜索

- 支持按文件名搜索。
- 支持按正文全文搜索。
- 搜索结果展示匹配文件、标题、片段。
- MVP 可使用 SQLite FTS 或轻量索引。

### 8.4 AI 侧边栏

- AI 侧边栏默认作用于当前笔记。
- 支持流式响应。
- 支持中断当前生成。
- 支持复制 AI 输出。
- 支持将 AI 输出插入到笔记。
- 写入类操作必须先展示预览。

### 8.5 AI 操作

MVP 内置以下 AI 动作：

- 总结当前笔记。
- 提取待办事项。
- 改写选中文本。
- 压缩选中文本。
- 扩写选中文本。
- 建议标题。
- 建议标签。
- 提出可改进建议。

### 8.6 设置

- 配置 API Key。
- 选择模型。
- 配置 AI 是否允许读取当前笔记、关联笔记或全库索引。
- 配置自动保存策略。
- 查看当前 vault 路径。

## 9. 非功能需求

### 性能

- 打开应用到可交互时间应尽量控制在 2 秒内。
- 打开普通 Markdown 文件应接近即时。
- 大文件编辑时不能明显卡顿。
- 索引构建应后台执行，并显示状态。

### 安全与隐私

- API Key 不应明文保存在普通配置文件中，应使用系统安全存储或 Tauri 推荐方案。
- AI 默认只读取当前笔记。
- 读取整个笔记库必须由用户显式授权。
- AI 写入必须由用户确认。
- 错误日志不应记录完整笔记正文或 API Key。

### 稳定性

- 文件写入失败必须保留用户编辑内容。
- 应支持基础快照或修改前备份。
- 外部工具修改文件后，应用应检测变化并提示用户。

## 10. 数据模型建议

Markdown 文件保存在用户选择的本地目录中。SQLite 只保存索引和元数据，不作为笔记正文的唯一来源。

SQLite 可包含：

- vaults：已打开笔记库列表。
- notes_index：文件路径、标题、更新时间、摘要、正文索引状态。
- tags：标签列表。
- note_tags：笔记和标签关系。
- ai_cache：摘要、标签建议、最近 AI 操作结果。
- snapshots：修改前快照或恢复点。

## 11. AI 设计原则

1. AI 默认是助手，不是自动管理员。
2. AI 默认读取当前笔记，不默认读取全库。
3. AI 输出和写入分离。
4. 写入前必须可预览。
5. 用户可以撤销 AI 修改。
6. Prompt 应包含明确边界，避免 AI 编造文件内容。
7. 对“当前笔记外”的上下文，AI 必须明确说明依据来源。

## 12. 推荐系统提示词方向

AI 助手应遵守以下行为边界：

- 只根据提供的笔记内容回答，不假装读过未提供的文件。
- 对不确定内容明确说明不确定。
- 输出尽量结构化、简洁、可直接使用。
- 修改笔记时保留用户原意和语气。
- 不主动删除重要内容。
- 不在未经用户确认时写入文件。
- 生成标签时优先使用短标签、稳定标签、可复用标签。
- 总结时区分事实、观点、待办和开放问题。

## 13. 里程碑

### M1：本地 Markdown MVP

- Tauri 应用可启动。
- 可选择 vault。
- 可浏览 Markdown 文件树。
- 可打开、编辑、保存 Markdown。
- 可基础搜索。

### M2：当前笔记 AI

- 可配置 API Key 和模型。
- AI 侧边栏可总结当前笔记。
- 可提取待办。
- 可改写选中文本。
- 写入前展示预览。

### M3：索引与标签

- SQLite 索引稳定。
- 支持标签建议。
- 支持摘要缓存。
- 支持最近打开和状态恢复。

### M4：全库问答探索

- 用户授权后建立全库检索。
- 支持“问我的笔记”。
- 回答必须展示引用来源。

## 14. 成功标准

MVP 成功标准：

- 用户能在 5 分钟内打开自己的 Markdown 文件夹并开始编辑。
- 用户能对当前笔记生成有用摘要。
- 用户能放心让 AI 改写文本，因为所有写入前都有预览。
- 应用不会破坏现有 Markdown 文件结构。
- 不依赖 Python 也能完成第一版核心体验。

## 15. 风险与对策

### 风险：AI 错误修改笔记

对策：所有写入都必须预览、确认、可撤销。

### 风险：文件和数据库状态不一致

对策：Markdown 文件是事实来源，数据库只做索引和缓存。

### 风险：过早加入复杂架构

对策：MVP 只支持一个主要 AI provider、当前笔记上下文、有限内置动作。

### 风险：跨平台文件权限复杂

对策：第一阶段优先 macOS，架构上保留 Windows/Linux 兼容空间。

### 风险：AI 成本不可控

对策：默认只发送当前笔记，显示输入范围，后续加入 token 估算和调用历史。

## 16. 下一步建议

下一轮实现前，应先输出技术设计和实现计划，而不是直接生成代码。建议顺序：

1. 确定项目名、包名和目标平台。
2. 初始化 Tauri + React + TypeScript 项目。
3. 设计 Rust command 边界。
4. 设计前端页面结构和状态管理。
5. 先实现本地 Markdown 文件能力。
6. 再接入 AI 侧边栏。
7. 最后加入 SQLite 索引和搜索增强。
